Configuration

config/config.yaml — single source of truth

YAML — config/config.yaml
# Browser settings
browser:
  name: chrome       # firefox, edge, etc.
  headless: false    # true for CI
  timeout: 15
  implicit_wait: 10

base_url: https://www.purplle.com

urls:
  wishlist: https://www.purplle.com/profile/myfavourites
  cart:     https://www.purplle.com/cart

api:
  endpoints:
    search:   /api/search
    products: /api/products
Robot Framework — using config
*** Settings ***
Library    ../../config/readme.py

*** Test Cases ***
Example Test
    ${url}=     Get Base Url
    ${timeout}= Get Timeout
    ${mobile}=  Get Test Credential  mobile
    Open Application
    # URL, timeout, credentials all from config.yaml
    # No hardcoded values in test files
ℹ️
Edit config, not tests
Change browser, timeouts, URLs or test data in config.yaml — no test file changes needed.

Test Layers

Four distinct layers with clear responsibilities

🔵
Functional Tests
tests/functional/
Tests a single module in isolation. Each file maps 1-to-1 with a page or feature. Imports 1–2 page resources.
test_homepage.robot — Title, header, nav links
test_login.robot — Form presence, OTP validation
test_search.robot — Valid/invalid searches, suggestions
test_product_listing.robot — Products, sort, filters
test_product_detail.robot — Name, price, Add to Bag
test_cart.robot — Empty state, cart icon
test_wishlist.robot — Empty state, /profile/myfavourites
test_navigation.robot — All nav links, logo, back
test_datadriven_search.robot — 8 CSV keywords
🟣
Integration Tests
tests/integration/
Tests how two or more modules work together in a single browser flow. Imports 2+ page resources.
test_search_to_pdp_flow — Search → PDP → Back
test_plp_to_pdp_flow — Category PLP → PDP → Brand
test_homepage_to_category_flow — Home nav → PLP → Cart
test_search_filter_flow — Search → Sort → Filter → PDP
🌸
E2E Journey Tests
tests/e2e/
Complete realistic user journeys from start to finish. Imports all relevant page resources.
test_guest_browse_journey — Home → Category → PDP → Cart → Wishlist
test_search_and_view_journey — Search → Sort → PDP → Compare
test_category_navigation_journey — Multi-category → Filter → Search → Brands
🟢
API Tests
tests/api/
HTTP-level behaviour using RequestsLibrary. No browser needed. Fastest layer.
test_search_api — Status codes, JSON response, no 5xx
test_product_api — Products, categories, HTTPS enforcement, response time
⚠️
API endpoint paths are best-guess patterns. Verify from Chrome DevTools → Network tab before running.

Running Tests

Commands to execute different test configurations

PowerShell / Bash
# Run an entire test layer
robot -d outputs tests/functional/
robot -d outputs tests/integration/
robot -d outputs tests/e2e/
robot -d outputs tests/api/

# Run ALL tests
robot -d outputs tests/

# Run in parallel (pabot)
pabot tests/
pabot --processes 4 tests/functional/
PowerShell / Bash
robot -d outputs tests/functional/test_homepage.robot
robot -d outputs tests/functional/test_datadriven_search.robot
robot -d outputs tests/functional/test_login.robot
robot -d outputs tests/integration/test_search_to_pdp_flow.robot
robot -d outputs tests/api/test_search_api.robot
PowerShell / Bash — use --include, not --tag
robot -d outputs --include smoke tests/          # Quick checks before deploy
robot -d outputs --include regression tests/      # Full regression suite
robot -d outputs --include functional tests/      # All functional tests
robot -d outputs --include integration tests/     # All integration tests
robot -d outputs --include e2e tests/             # All E2E journey tests
robot -d outputs --include api tests/             # All API tests
robot -d outputs --include data_driven tests/     # DataDriver CSV tests
robot -d outputs --include negative tests/         # Error-path tests
Special Modes
# Headless mode — required on Linux CI / no display
robot -d outputs -v HEADLESS:true tests/
robot -d outputs -v HEADLESS:true tests/functional/test_homepage.robot

# Dry run — validate keywords without launching browser
robot --dryrun tests/
robot --dryrun tests/functional/test_datadriven_search.robot

# Install dependencies
pip install -r requirements.txt

# Open reports after run
start outputs\report.html       # Windows
open  outputs/report.html       # macOS
xdg-open outputs/report.html  # Linux

Data-Driven Search Flow

DataDriver generates 8 test cases from one CSV file

${search_keyword} ← RF variable format required
1lipstick✓ PASS
2foundation✓ PASS
3moisturizer✓ PASS
4serum✓ PASS
5shampoo✓ PASS
6perfume✓ PASS
7kajal✓ PASS
8sunscreen✓ PASS
DataDriver reads CSV → generates 8 test cases at runtime.
Each test navigates to /search?q=${keyword} and waits for product cards.
test_datadriven_search.robot (pattern)
*** Settings ***
Library      DataDriver
...          file=../../testdata/purplle_search_keywords.csv
Test Template    Search And Verify By Keyword

*** Test Cases ***
Search For ${search_keyword}    # template row

*** Keywords ***
Search And Verify By Keyword
    [Arguments]    ${keyword}
    Go To    ${BASE_URL}/search?q=${keyword}
    Wait Until Element Is Visible
    ...    xpath=(//a[contains(@class,'d-block') and contains(@href,'/product/')])[1]
    ...    timeout=${TIMEOUT}
⚠️
RF 7 Compatibility
CSV column header must use ${variable} syntax. Template keyword must be in the test file's own *** Keywords *** section — DataDriver cannot find keywords from imported Resource files at listener init time.

Key Conventions

Rules that keep the framework consistent and maintainable

📁
Locators stay in locators/
Always go in locators/*_locators.robot — never inline in keywords or test cases. Keeps test files readable and locators centralized.
⏱️
Wait Until Element Is Visible — never Sleep
Use Wait Until Element Is Visible with explicit timeout (15s default). Sleep is fragile — it either wastes time or fails on slow networks.
🏷️
Test ID Format: TC_<LAYER>_<MODULE>_<NN>
e.g. TC_FUNC_HOME_01, TC_INT_FILTER_03, TC_E2E_GUEST_02, TC_API_SEARCH_06
🔐
Guest-only flows in CI
Purplle uses phone OTP login which cannot be automated. Tests requiring login are tagged requires_login and skipped in CI.
🛍️
"ADD TO BAG" not "Add to Cart"
Purplle's UI uses this specific label. All locators in product_detail_page_locators.robot reflect the actual site text.
📍
Resource import paths
Tests in tests/functional/, integration/, e2e/ use ../../resources/ and ../../locators/. Legacy tests in tests/ root use ../resources/.
✉️
Tag every test
Each test must have at least: layer tag (functional/integration/e2e/api) + feature tag + either smoke or regression.
